# ChatForYou v2: 설계 가이드 (표준 01-06)

이 가이드는 프로젝트의 모든 기능 개발 전 단계에서 수행해야 하는 **'분석 및 설계 표준'**을 정의합니다. 모든 참여자(AI 및 개발자)는 구현 코드를 작성하기 전, 본 가이드에 따라 `plan_docs/` 내에 문서를 먼저 작성해야 합니다.

## 1. 핵심 원칙
- **설계 우선**: 코드를 수정하기 전, 반드시 설계 문서를 통해 논리적 결함이 없는지 검토한다.
- **코드 금지 원칙**: "개발해줘", "구현해줘"라는 명시적 요청이 없는 한, 소스 코드(`.java`, `.js`, `.desktop`)를 수정하지 않고 설계 문서 작성에만 집중한다.
- **계층화 설계**: 모든 설계는 정보 계층(L1~L4)에 따라 구체화한다.
- **위험도 비례 프로세스**: 변경의 위험도(5절 참고)에 따라 필요한 Phase 범위가 결정된다. 모든 변경에 전체 사이클을 적용하지 않는다.

## 2. 디렉토리 구조 (01-06 단계)

### [기본 계획] MM월_[기능]_plan.md
- 모든 기능 분석의 시작점. 논의 과정, 결론, 유의점, 관련 문서 맵핑을 기록한다.
- **문서 매핑 필수 항목**: 기본 계획의 `## Document Mapping` 섹션에는 아래 항목을 반드시 포함해야 한다.
  ```
  - [ ] 요구사항 및 데이터: `plan_docs/01-plan/[feature].md`
  - [ ] 인터페이스 및 시퀀스: `plan_docs/02-design/[feature]-spec.md`
  - [ ] 구현 가이드: `plan_docs/03-implementation/[feature]-guide.md`
  - [ ] 백엔드 구현 가이드: `springboot-backend/plan_docs/[feature].md`
  - [ ] 프론트 구현 가이드: `nodejs-frontend/plan_docs/[feature].md`
  - [ ] 갭 분석: `plan_docs/04-analyze/[feature]-gap.md`
  - [ ] 전문가 리뷰: `plan_docs/05-expert-review/[feature]-review.md`
  - [ ] 최종 보고서: `plan_docs/06-report/[feature].md`
  ```
- **확인 규칙**: 체크박스 상태([ ] / [x])와 무관하게, 작업 시작 전 나열된 모든 경로를 파일시스템에서 직접 존재 여부를 확인하고 존재하는 파일은 반드시 읽어야 한다.

### 01-plan (L1/L2: 컨텍스트 및 도메인)
- **L1 (시스템 컨텍스트)**: 기능이 프로젝트 전체(Backend, Frontend, Desktop) 중 어디에 속하며 외부 시스템과 어떻게 연결되는지 정의.
- **L2 (데이터 모델)**: 필요한 DB 테이블 구조, Entity 필드, JWT 클레임 등 데이터 계층 정의.

### 02-design (L3: 인터페이스 및 시퀀스)
- **L3 (인터페이스)**: API 엔드포인트(Method, Path, Request/Response) 상세 정의.
- **시퀀스 다이어그램**: 플랫폼 간(예: Electron ↔ Backend) 데이터 흐름과 상태 변화를 시각화.
- **검증 시나리오 (필수)**: `## 3. 검증 시나리오` 섹션에 전제/실행/기대(Given/When/Then) 형식으로 핵심 시나리오를 기술한다. QA 전문가가 이 시나리오를 기준으로 테스트를 작성한다.

### 03-implementation (L4: 구현 전략)
- **L4 (상세)**: 실제 구현 시 수정해야 할 파일 목록, 로직의 핵심 알고리즘, 환경 설정(Config) 변경 사항 기록.

### 04-analyze (갭 분석)
- 구현 완료 후, 실제 코드와 01-03 설계 문서의 차이를 분석한다.
- **범위**: 설계 대비 구현 차이(Match Rate), 누락 항목, 리스크, 버그 패턴 탐지.
- 코드 품질·아키텍처·보안 평가는 이 단계에서 수행하지 않는다 → 05-expert-review 담당.

### 05-expert-review (전문가 리뷰)
- 외부 전문가 관점의 코드 품질 평가.
- **범위**: 코드 품질, 아키텍처 적합성, 보안 취약점, 성능, 운영 관점 평가.
- 갭 재분석은 이 단계에서 수행하지 않는다 → 04-analyze 담당.

### 06-report (최종 보고)
- 기능 완성 보고, 테스트 결과물, 향후 개선 사항 기록.

## 3. 외부 전문가 워크플로우
1. **분석 단계**: `plan_docs/` 하위에 01, 02, 03 파일을 생성하고 설계한다.
2. **세분화 단계**: 각 모듈(`springboot-backend/`, `nodejs-frontend/`) 내부의 `plan_docs/`에 더 구체적인 기술 가이드를 작성한다.
3. **구현 단계**: 설계가 승인된 후 코딩을 시작한다.
4. **빌드 및 테스트 검증**: 구현 완료 직후, 반드시 로컬 환경에서 전체 빌드와 테스트를 수행하여 성공(Success) 여부를 확인한다. 빌드 실패 시 해당 단계는 완료된 것으로 간주하지 않는다.
5. **갭 분석**: 구현 완료 후 04-analyze 문서를 작성한다 (설계 대비 구현 차이 분석).
6. **전문가 리뷰**: 위험도에 따라 05-expert-review 문서를 작성한다 (품질·보안·운영 평가).
7. **최종 보고**: 06-report 문서로 사이클을 마무리한다.

## 4. 문서 품질 기준
- "다른 AI가 이 문서를 읽고 즉시 에러 없이 코딩할 수 있을 정도"로 구체적이어야 한다.
- 추상적인 설명보다는 구체적인 JSON 구조, 메서드 명, 파일 경로를 사용한다.

## 5. 변경 위험도 분류

모든 변경 요청은 아래 4단계 중 하나로 분류하고, 필요한 Phase 범위를 결정한다.

| 단계 | 분류 | 필수 Phase | 비고 |
|---|---|---|---|
| L0 | 문서·텍스트 변경만 | 없음 | 요약 + 변경 파일 목록만 작성. 코드 변경 없음. |
| L1 | UI / 비핵심 로직 | 00~03 | 동작 변경이 수반되는 경우 04 선택적 추가. |
| L2 | 백엔드 상태 / Redis / 인증 | 00~05 | 빌드·테스트 증거 필수. |
| L3 | WebRTC / WebSocket / 시그널링 / 데스크탑 동기화 | 00~06 | 설계 리뷰 최소 2회 필수. 리뷰 문서화 완료 전 구현 차단. |
